注意，所有的版本控制系统(Version control system，例如Git)只能记录文本文件(例如txt文件，代码文件)的改动，不能够记录Word啊，图片啊之类的改动。其中文本文件最好都使用utf-8的编码格式，因为所用平台都是支持的。

1.5 Git的工作区与暂存区

Git 是面向修改的管理，而不是面向文件的管理，也就是说每一次我们做了修改以后，需要将修改add到暂存区，在暂存区的修改点击commit才到我们的master分支，也就是将暂存区的所有修改都进行提交。如果仅仅修改文件，而不add到缓存区，即使commmit也不会有任何的好处。

1.6 Git的撤销修改

当我们对文件做了一次修改，例如我在readme.txt文件中增加一行I am a fool，此时我还没有add或者此时我add了但又做了修改。然后我忽然想要删除这句话。当然我们可以手动的修改回去，我们也可以使用Vim的u指令回跳，但是当我们做了几百行的修改这样就会很耗时间。Git中提供了一个简单的方法checkout。

checkout会使我们的文件到达最近的一次commit或者add的状态，记住这里是或者，看离你checkout最近的指令是哪个。

3.3 分支管理策略

在团队开发当中，很少像我们这样，每一次的分支到最后都会合并到master分支上，相反，我们习惯于建立下面几个分支：

  • master分支，用于版本的发布
  • dev分支，用于真正的团队开发分支，完成一次全面的开发后合并到master分支
  • yourname分支，这个使得每一个组员都应有的一个分支，在自己的分支上做开发，并且合并到dev分支上

5. 相关指令总结

指令 效果
git init 初始化一个respondary
git add 在仓库里增加一个文件
git commit 提交更改
git status 查看状态
git diff 查看不同
git log 查看日志
git reset 重设到之前的状态
git reflog 查看自己的指令
git remote add origin git@github.com:ZhaochengD/Learn_Git 远程连接我的库
git push -u origin master 首次推送库中内容，以后可以省略-u
git clone 从远程上克隆库到本地
git branch dev 创建一个分支dev，-d是删除，什么也不加是列出所有分支，-D强制删除
git checkout dev 开始checkout分支
git merge dev 合并dev分支到master
git remote 查看远程仓库
git push 将分支的修改推送到远程
